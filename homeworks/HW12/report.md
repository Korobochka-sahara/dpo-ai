# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Использовался датасет `S12-hw-dataset.csv`, прогнозировалась колонка `target`.
- Данные были отсортированы по времени и разбиты на `train / validation / test` в пропорции 70/15/15.
- Были реализованы и сравнены подходы: `B1 (naive-last)`, `B2 (moving-average)`, `B3 (ridge-lag-features)`, `R1 (gru-forecast)`.
- Лучший результат показала модель `R1 (GRU)` благодаря способности учитывать последовательную зависимость во временном ряде.

## 2. Среда и воспроизводимость

- Python: 3.x
- pandas / numpy / scikit-learn: актуальные версии
- torch: актуальная версия
- Устройство (CPU/GPU): CPU (или GPU при наличии)
- Seed: 42
- Как запустить: открыть `HW12.ipynb` и выполнить Run All

## 3. Данные и постановка задачи

- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: `target`
- Частота временного ряда: почасовая (hourly)
- Горизонт прогноза: 1 шаг вперёд
- Размер датасета: ~N наблюдений
- Диапазон дат: от начала до конца датасета
- Есть ли дополнительные внешние признаки: нет (использовались только производные от target)

Комментарий:
Временной ряд демонстрирует признаки тренда и возможной сезонности. Значения изменяются плавно, но присутствуют локальные колебания. Ряд может быть нестационарным, что усложняет задачу прогноза. Пропусков после обработки лагов не осталось (строки с NaN были удалены).

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Использовано разбиение: первые 70% — train, следующие 15% — validation, последние 15% — test
- Разбиение выполнено строго по времени без перемешивания
- `random split` некорректен, так как приводит к утечке информации из будущего и завышенным метрикам

### 4.2. Признаки для baseline-моделей

- Лаги: `lag_1`, `lag_7`, `lag_14`
- Rolling-признаки: `rolling_mean_7`, `rolling_std_7`
- Календарные признаки: `day_of_week`
- Пропуски после генерации признаков удалялись (`dropna`)
- Масштабирование не использовалось

## 5. Модели и эксперименты (B1, B2, B3, R1)

- B1 (`naive-last`): прогноз равен предыдущему значению
- B2 (`moving-average`): скользящее среднее по окну 7
- B3 (`ridge-lag-features`): линейная модель Ridge на лаговых и календарных признаках
- R1 (`gru-forecast`): рекуррентная модель GRU на оконном представлении ряда

Дополнительно:

- Основная метрика выбора: `MAE`
- Дополнительные метрики: `RMSE`, `MAPE`
- `window_size`: 24
- Batch size: 32
- Максимальное число эпох: 10
- Optimizer: Adam
- Learning rate: 1e-3
- Лучшая модель сохранялась по минимальному `val_MAE`

- Лучшая `val_RMSE`: 8.21 (B1)

- GRU показала существенно более высокую ошибку (Val MAE = 95.32) по сравнению с табличным baseline (B3), что указывает на избыточность нейросетевой архитектуры для данного ряда или необходимость более тщательной настройки гиперпараметров

- Выводы по validation и test согласуются: лучшей моделью на валидации является B3, и на тесте она также показывает наилучшие результаты среди всех подходов

## 7. Анализ

Наивный baseline оказался слабым, так как не учитывает ни сезонность, ни тренд. 
Moving average сглаживает ряд, но теряет важные изменения и не реагирует на резкие колебания. 
Добавление лаговых признаков позволило модели Ridge учитывать временные зависимости, что значительно улучшило качество.

GRU показала существенно более высокую ошибку (MAE 95.32) по сравнению с Ridge (MAE 7.17). Это может быть связано с:
- Недостаточным объемом данных для обучения нейросети
- Неоптимальными гиперпараметрами (window_size=24, hidden_size=32)
- Отсутствием масштабирования данных
- Простой линейной структурой временного ряда, которая лучше описывается линейной моделью

В данных заметна некоторая сезонность, что объясняет эффективность лагов (например lag_7). 
Возможные утечки данных могли бы возникнуть при использовании будущих значений в признаках или при неправильном scaling.

Выбор модели по validation критичен, так как test должен использоваться только для финальной оценки.

## 8. Итоговый вывод

Лучшим подходом на данном датасете оказалась Ridge регрессия на лаговых признаках - 
она проста и даёт стабильный результат с MAE = 7.17.

GRU (рекуррентная нейросеть) показала значительно более высокую ошибку (MAE = 95.32), 
что говорит о том, что для данного ряда сложные нейросетевые архитектуры могут быть 
избыточны или требуют более тщательной настройки гиперпараметров.

Для простых задач достаточно baseline-моделей. Нейросети оправданы при наличии сложной 
структуры данных и достаточного объема обучающей выборки.

Главный вывод - для временных рядов критически важно корректное разбиение по времени, 
иначе метрики становятся недостоверными.
